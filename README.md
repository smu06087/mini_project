6-1.조회수 중복 증가 문제
🔹 문제
게시글 상세 페이지에서 새로고침(F5) 또는 동일 사용자의 반복 접근 시
조회수가 계속 증가하는 문제가 발생했습니다.
이로 인해 실제 조회수보다 과대 집계되는 현상이 나타났습니다.

🔹 원인
조회수 증가 로직이 페이지 접근 시마다 실행되는 구조였으며,
사용자의 중복 접근 여부를 검증하지 않은 상태였습니다.
즉, 요청(Request) 단위로 조회수를 증가시키는 단순 처리 방식이 문제였습니다.

🔹 해결
세션(Session)을 활용하여 중복 조회를 방지하는 로직을 추가했습니다.
사용자가 조회한 article_no를 세션에 저장
동일 세션에서 동일 게시글 재접속 시 조회수 증가 로직 미실행
조회수 증가 조건을 분기 처리하여 제어

🔹 결과
중복 조회수 증가 문제 해결
보다 신뢰성 있는 조회수 집계 가능
상태 관리 기반 로직 설계 경험 확보

🔹 배운 점
웹 애플리케이션에서는 단순 요청 처리보다
사용자 상태(Session) 기반의 로직 설계가 중요함을 학습했습니다.
특히, 데이터 무결성을 유지하기 위한 조건 제어의 필요성을 경험했습니다.


6-2.댓글 계층 구조 출력 문제
🔹 문제
게시글에 대댓글 기능을 추가하는 과정에서
댓글이 부모-자식 구조로 올바르게 정렬되지 않는 문제가 발생했습니다.
단순 정렬 방식으로는 계층 구조 표현이 불가능했습니다.

🔹 원인
댓글 테이블은 parent_comment_id를 기반으로 계층 구조를 설계했으나,
초기 구현에서는 단순 반복문 출력 방식으로 처리하여
부모-자식 관계가 반영되지 않았습니다.

🔹 해결
재귀(Recursive) 함수 기반 출력 구조로 변경했습니다.
최상위 댓글을 먼저 출력
해당 댓글을 부모로 갖는 자식 댓글을 재귀적으로 호출
depth(레벨) 값을 활용하여 들여쓰기 처리
이를 통해 트리 구조 기반 출력이 가능하도록 개선했습니다.

🔹 결과
댓글 계층 구조 정상 출력
대댓글 무제한 확장 가능 구조 확보
데이터 구조에 맞는 출력 로직 구현

🔹 배운 점
계층형 데이터는 단순 정렬이 아닌
트리 구조 관점에서 접근해야 한다는 점을 학습했습니다.
자료구조와 로직 설계의 중요성을 경험한 사례였습니다.


6-3. 작성 권한 및 URL 직접 접근 문제
🔹 문제
비로그인 사용자가 URL을 직접 입력하여
글쓰기 페이지에 접근할 수 있는 문제가 발생했습니다.
또한, 작성자가 아닌 사용자가 수정/삭제 요청을 시도할 수 있는 구조적 취약점이 존재했습니다.

🔹 원인
프론트엔드에서 버튼을 숨기는 방식으로만 접근을 제어했으며,
서버 단에서 세션 및 작성자 검증 로직이 충분히 구현되지 않았습니다.

🔹 해결
서버단에서 세션 memberId 존재 여부 검증
글 수정/삭제 시 작성자와 세션 사용자 비교
조건 불일치 시 접근 차단 및 리다이렉트 처리
즉, UI 제어가 아닌 서버단 권한 검증 로직을 추가하여 보안을 강화했습니다.

🔹 결과
비인가 사용자 접근 차단
작성자 권한 기반 기능 제어 가능
보안 및 무결성 강화

🔹 배운 점
권한 제어는 화면 단에서 숨기는 것이 아니라
반드시 서버단에서 검증되어야 한다는 점을 학습했습니다.
백엔드에서의 접근 제어 설계가 중요함을 경험했습니다.
