4. 시스템 구조 (Architecture) <br>
본 프로젝트는 MVC(Model-View-Controller) 패턴을 기반으로 설계되었습니다. <br>
역할을 분리하여 요청 처리 흐름을 명확히 하고, 유지보수성과 확장성을 고려한 구조로 구현하였습니다. <br>

전체 요청 처리 흐름 <br>
Client → Controller(Servlet) → DAO → DB  
　　　　　　　　　　　↓  
　　　　　　　　　　JSP(View)

🔹 Controller (Servlet) <br>
- 클라이언트 요청 처리 및 URL 매핑 <br>
- 요청 파라미터 수집 및 DAO 호출 <br>
- 세션(Session)을 활용한 로그인 상태 및 권한 검증 <br>
- 처리 결과를 JSP(View)로 전달 <br>

🔹 DAO (Data Access Object) <br>
- Oracle DB 접근 및 SQL 실행 <br>
- 게시글(Article), 댓글(Comment), 회원(Member), 카테고리(Category) 데이터 CRUD 처리 <br>
- 비즈니스 로직과 데이터 접근 로직 분리 <br>

🔹 View (JSP) <br>
- EL, JSTL을 활용한 데이터 출력 <br>
- 조건문을 통한 권한 기반 UI 제어 <br> 
- 댓글 계층 구조 재귀 출력 구현 <br>
- 작성자 본인 여부에 따른 수정/삭제 버튼 분기 처리 <br>

구현 포인트 <br>
- 게시판 및 댓글 기능은 JSP/Servlet 구조에서 직접 구현 <br>
- 요청 처리 흐름을 이해하고 Controller → DAO → View 흐름을 명확히 구성 <br>
- 세션 기반 권한 제어 및 조회수 중복 방지 로직 설계 <br>
- 단순 CRUD를 넘어 상태(Session) 기반 로직을 적용하여 데이터 무결성 확보 <br>

6. 트러블 슈팅 (Trobuleshooting) <br>

6-1. 조회수 중복 증가 문제 <br>
🔹 문제 <br>
게시글 상세 페이지에서 새로고침(F5) 또는 동일 사용자의 반복 접근 시 조회수가 계속 증가하는 문제가 발생했습니다. <br>
이로 인해 실제 조회수보다 과대 집계되는 현상이 나타났습니다. <br>

🔹 원인 <br>
조회수 증가 로직이 페이지 접근 시마다 실행되는 구조였으며, 사용자의 중복 접근 여부를 검증하지 않은 상태였습니다. <br>
즉, 요청(Request) 단위로 조회수를 증가시키는 단순 처리 방식이 문제였습니다.

🔹 해결 <br>
세션(Session)을 활용하여 중복 조회를 방지하는 로직을 추가했습니다. <br>
사용자가 조회한 article_no를 세션에 저장 <br>
동일 세션에서 동일 게시글 재접속 시 조회수 증가 로직 미실행 <br>
조회수 증가 조건을 분기 처리하여 제어 <br>

🔹 결과 <br>
중복 조회수 증가 문제 해결 <br>
보다 신뢰성 있는 조회수 집계 가능 <br>
상태 관리 기반 로직 설계 경험 확보 <br>

🔹 배운 점 <br>
웹 애플리케이션에서는 단순 요청 처리보다 사용자 상태(Session) 기반의 로직 설계가 중요함을 학습했습니다. <br>
특히, 데이터 무결성을 유지하기 위한 조건 제어의 필요성을 경험했습니다. <br>

6-2. 댓글 계층 구조 출력 문제 <br>
🔹 문제 <br>
게시글에 대댓글 기능을 추가하는 과정에서 댓글이 부모-자식 구조로 올바르게 정렬되지 않는 문제가 발생했습니다. <br>
단순 정렬 방식으로는 계층 구조 표현이 불가능했습니다. <br>

🔹 원인 <br>
댓글 테이블은 parent_comment_id를 기반으로 계층 구조를 설계했으나, 초기 구현에서는 단순 반복문 출력 방식으로 처리하여 <br>
부모-자식 관계가 반영되지 않았습니다. <br>

🔹 해결 <br>
재귀(Recursive) 함수 기반 출력 구조로 변경했습니다. <br>
최상위 댓글을 먼저 출력 <br>
해당 댓글을 부모로 갖는 자식 댓글을 재귀적으로 호출 <br>
depth(레벨) 값을 활용하여 들여쓰기 처리 <br>
이를 통해 트리 구조 기반 출력이 가능하도록 개선했습니다. <br>

🔹 결과 <br>
댓글 계층 구조 정상 출력 <br>
대댓글 무제한 확장 가능 구조 확보 <br>
데이터 구조에 맞는 출력 로직 구현 <br>

🔹 배운 점 <br>
계층형 데이터는 단순 정렬이 아닌 트리 구조 관점에서 접근해야 한다는 점을 학습했습니다. <br>
자료구조와 로직 설계의 중요성을 경험한 사례였습니다. <br>

6-3. 작성 권한 및 URL 직접 접근 문제 <br>
🔹 문제 <br>
비로그인 사용자가 URL을 직접 입력하여 글쓰기 페이지에 접근할 수 있는 문제가 발생했습니다. <br>
또한, 작성자가 아닌 사용자가 수정/삭제 요청을 시도할 수 있는 구조적 취약점이 존재했습니다. <br>

🔹 원인 <br>
프론트엔드에서 버튼을 숨기는 방식으로만 접근을 제어했으며, 서버 단에서 세션 및 작성자 검증 로직이 충분히 구현되지 않았습니다. <br>

🔹 해결 <br>
서버단에서 세션 memberId 존재 여부 검증 <br>
글 수정/삭제 시 작성자와 세션 사용자 비교 <br>
조건 불일치 시 접근 차단 및 리다이렉트 처리 <br>
즉, UI 제어가 아닌 서버단 권한 검증 로직을 추가하여 보안을 강화했습니다. <br>

🔹 결과 <br>
비인가 사용자 접근 차단 <br>
작성자 권한 기반 기능 제어 가능 <br>
보안 및 무결성 강화 <br>

🔹 배운 점 <br>
권한 제어는 화면 단에서 숨기는 것이 아니라 반드시 서버단에서 검증되어야 한다는 점을 학습했습니다. <br>
백엔드에서의 접근 제어 설계가 중요함을 경험했습니다. <br>
